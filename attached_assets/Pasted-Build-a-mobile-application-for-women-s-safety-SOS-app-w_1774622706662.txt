Build a mobile application for women’s safety (SOS app) with a Java-based backend using Spring Boot and a smooth, minimal, classy UI, where the app requests clear user permissions for microphone, location, motion sensors, camera, and background activity, and operates in a controlled background mode that the user can turn on or off (for example, active monitoring only during selected hours or when manually enabled to address privacy and battery concerns). The app should use the Google Gemini API for audio analysis but not rely on it alone; instead implement a multi-signal detection system that combines AI-based distress classification with keyword spotting (such as “help”, “bachao”, or similar distress phrases), contextual confidence thresholds, and motion anomalies to reduce false positives. When a potential threat is detected, the app should not immediately trigger but instead initiate a short verification phase that includes multiple confirmation options such as a specific shake pattern (not a single jerk), a voice confirmation keyword, or automatic escalation if there is no response within a few seconds, along with a brief vibration or on-screen alert allowing the user to cancel to prevent accidental triggers. Once confirmed (or auto-confirmed after timeout), the app should enter a stealth emergency mode where audio recording is guaranteed and video recording is attempted where supported, while displaying a black screen overlay or minimal UI to avoid drawing attention, with fallback handling in case background camera access is restricted. Simultaneously, the app should fetch real-time GPS location and continuously update it, then securely send an alert package containing live location, user profile details, and captured media to predefined trusted emergency contacts (primary requirement), and also to a simulated nearest police station endpoint by identifying the closest station using map services and displaying it in the system (even if actual government integration is mocked for the prototype). All emergency data including media, timestamps, and location logs should be uploaded securely to cloud storage (such as Firebase or AWS) with proper encryption and access control. The backend should be implemented in Java using Spring Boot with RESTful APIs for user management, media uploads, alert triggering, and contact notification, optionally secured with JWT authentication, and designed to handle real-time data flow efficiently. The frontend should be built using Flutter or React Native with a clean onboarding flow for permissions and emergency contact setup, and include simple screens such as home status (safe/monitoring), emergency contacts management, and settings, ensuring the overall experience is fast, intuitive, and non-intrusive while maintaining transparency about data usage and giving users full control over when monitoring is active.
